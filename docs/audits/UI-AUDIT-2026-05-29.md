# Glasses UI Audit — Hermes ↔ G2 Client

**Date:** 2026-05-29
**Scope:** The glasses-display UI implemented so far, audited against the Even Hub `glasses-ui` guidelines (576×288 canvas, container rules, page-lifecycle methods, BLE serialization best practices).
**Files reviewed:** `src/ui/render.ts`, `src/ui/views.ts`, `src/main.ts`, `src/input/router.ts`, `src/state/store.ts`.
**Related:** `PLAN-REVIEW-2026-05-29.md` (plan-level review). Overlaps are cross-referenced; this doc covers the **code that actually exists**, not the plan.

This is a findings document — no source files were modified.

## Status of the UI

The chat view is implemented and wired end-to-end (boot → build page → render on WS messages → route input). The sessions view (M3), voice (M4), and render throttling (Task 2.1) are **not yet implemented**. The audit reflects that reality: several findings are "implemented but unsafe," and a few are "not built yet, watch out when you do."

## ✅ Confirmed correct against the guidelines

- **Layout fits the canvas with no overlap.** header `y0 h40` (0–40), body `y44 h200` (44–244), status `y248 h36` (248–284). All within 576×288, 4px gutters, no overlapping regions.
- **Container budget respected.** `containerTotalNum: 3` matches exactly 3 `textObject`s; within the 8-text / 12-total limits.
- **Exactly one `isEventCapture: 1`** — the body container only. (Header and status correctly omit it.)
- **`paddingLength: 4`** is within the 0–32 range on every container.
- **Internal render ordering is correct** — `renderChat` `await`s its body `setText` before its status `setText`, so within a single render the two upgrades are serialized (the problem is *across* renders — see U1).
- **`store.reduce` clears chat on session switch** — `case "active"` resets `chat` to `initialState().chat`, so a switched-to session doesn't show the previous one's text.
- **`assistant` uses replace semantics** — the store stores the full accumulated text (`m.text`) rather than appending, matching the protocol's "client replaces" contract.

## 🔴 U1 — Streaming renders are fired concurrently (`main.ts`)

**Severity:** High — can crash the BLE connection under load.

**What the guideline says.** *"Serialize all bridge calls, not just images — `await` each before starting the next; concurrent render + storage calls can crash the connection."* And: use a coalescer for frequent updates.

**What the code does.** `main.ts:18`:

```ts
onMessage: (m) => { state = reduce(state, m); void renderChat(bridge, state); },
onStatus:  (s) => { state = { ...state, conn: s }; void renderChat(bridge, state); },
```

Every server frame fires `renderChat` with `void` (fire-and-forget, no await, no queue). During a streamed reply the bridge emits an `assistant` frame on **every token delta** — so dozens of `renderChat` calls overlap, each issuing two `textContainerUpgrade` BLE calls. They are neither serialized nor coalesced. This is exactly the "concurrent render calls" the guideline warns crashes the link, and it happens on the hot path (every streamed reply).

**Fix.** Implement Task 2.1 (it's planned but not built):
- A **render queue** that serializes calls — never start a `renderChat` while one is in flight; keep only the latest pending state.
- A **coalescer** (`requestAnimationFrame` or a short timeout window) so rapid `assistant` frames collapse to one render per window.

Sketch:

```ts
let rendering = false, pending: AppState | null = null;
function scheduleRender(bridge, s) {
  pending = s;
  if (rendering) return;
  rendering = true;
  queueMicrotask(async function flush() {
    while (pending) { const next = pending; pending = null; await renderChat(bridge, next); }
    rendering = false;
  });
}
```

**Verify.** In the simulator, prompt a long streamed reply and confirm the body updates smoothly without console errors or a dropped connection. On hardware, confirm no BLE stalls during streaming.

## 🔴 U2 — `textContainerUpgrade` omits `containerName` and replace params (`render.ts`)

**Severity:** High — documented silent-failure mode + stale text.

**What the guideline says.** Two explicit rules:
1. *"Always match `containerID` **and** `containerName` exactly when calling `textContainerUpgrade` — mismatches silently fail."*
2. *"Use `contentOffset: 0, contentLength: 0` for full content replacement."*

**What the code does.** `render.ts:20`:

```ts
await bridge.textContainerUpgrade(new TextContainerUpgrade({ containerID: id, content }));
```

No `containerName`, no `contentOffset`/`contentLength`. Per the guideline this risks a **silent no-op** (nothing renders, no error), and without explicit full-replace params, a shorter new string may not clear a longer previous one — leaving stale tail characters (very likely on the status line as it cycles `connecting…` → `connected` → `⚙ bash…` → `✓ done`, and on the body as the streamed tail shifts).

**Fix.** Thread the container name through `setText` and always pass full-replace params:

```ts
const NAMES = { [IDS.header]: "header", [IDS.body]: "body", [IDS.status]: "status" } as const;

export async function setText(bridge, id: number, content: string) {
  await bridge.textContainerUpgrade(new TextContainerUpgrade({
    containerID: id,
    containerName: NAMES[id],
    contentOffset: 0,
    contentLength: 0,   // full replacement
    content,
  }));
}
```

(Names must exactly match those used in `buildChatPage`.) Overlaps `PLAN-REVIEW` #6, but is more concrete now that the code exists.

**Verify.** Update the status line from a long string to a short one and confirm no leftover characters; confirm body updates actually appear on device.

## 🟡 U3 — Emoji glyphs may not render in the firmware font (`views.ts`)

**Severity:** Medium — silent visual breakage (tofu / blanks).

**What the guideline says.** *"Unicode is supported within the firmware font set."* That set is limited; full-color emoji are typically **not** included. (See the `design-guidelines` skill for icon/Unicode guidance.)

**What the code does.** `views.ts:11` builds status from `⚙` (gear) and `🎤`/`✓` appear in the plan. On a 4-bit greyscale display with a firmware font, `⚙` and `🎤` are emoji and may render as missing-glyph boxes or nothing. `✓` (U+2713) is more likely supported.

**Fix.** Verify each glyph on hardware/simulator. Prefer firmware-safe symbols — e.g. geometric/box-drawing characters (`●`, `▶`, `…`, `✓`) or short text labels (`run`, `done`) — over color emoji. Keep a single tested symbol set in one place.

**Verify.** Render each candidate glyph on device and screenshot; drop any that show as tofu.

## 🟡 U4 — No per-call timeout on BLE calls (`main.ts` / `render.ts`)

**Severity:** Medium.

**What the guideline says.** *"Add a per-call timeout to BLE calls — a single flaky hop can hang ~30s; wrap calls in `Promise.race` with a few-second cap."*

**What the code does.** `setText` / `buildChatPage` `await` SDK calls with no timeout. A flaky BLE hop can stall the render loop (and, once U1's queue exists, block all subsequent renders behind it).

**Fix.** Wrap bridge calls in a timeout helper:

```ts
const withTimeout = <T>(p: Promise<T>, ms = 4000) =>
  Promise.race([p, new Promise<never>((_, r) => setTimeout(() => r(new Error("ble timeout")), ms))]);
```

Apply inside `setText` (and the future image/storage calls). Pair with the render queue so a timed-out call doesn't wedge the pipeline.

## 🟡 U5 — Event-listener cleanup is discarded (`main.ts`)

**Severity:** Medium now, **High at M4** (mic left running).

**What the guideline / device-features say.** `onEvenHubEvent` returns an unsubscribe function; always unsubscribe on teardown, and stop hardware (`audioControl(false)`) on exit or the mic/IMU keeps running.

**What the code does.** `main.ts:23` calls `bridge.onEvenHubEvent(...)` and **discards the returned unsubscribe**. There is no `beforeunload`/foreground-exit teardown. Harmless in M1 (no hardware on), but once M4 adds `audioControl(true)`, exiting without cleanup leaves the microphone live.

**Fix.** Capture the unsubscribe and add teardown now, before audio lands:

```ts
const off = bridge.onEvenHubEvent(...);
window.addEventListener("beforeunload", () => { off(); /* M4: bridge.audioControl(false) */ });
```

Also route `FOREGROUND_EXIT_EVENT` (currently unhandled — see U6 / `PLAN-REVIEW` #7) to the same cleanup.

## 🟡 U6 — Scroll events are no-ops; long replies are unreadable (`main.ts` / `views.ts`)

**Severity:** Medium — functional gap.

**What the code does.** `router.ts` correctly maps `SCROLL_TOP_EVENT`/`SCROLL_BOTTOM_EVENT` to `onScrollUp`/`onScrollDown`, but `main.ts:26` wires both to `() => {}`. `renderChat` only ever shows `assistant.slice(-BODY_MAX)` (the **tail**). So any reply longer than one body-full can never be scrolled back to read — the earlier text is gone from view permanently.

**Fix.** Add a scroll offset to the chat state and paginate the body around it:
- `onScrollUp`/`onScrollDown` adjust a `bodyOffset` (in ~`BODY_MAX`-char or `\n`-aware pages).
- `renderChat` slices the visible window from `bodyOffset` instead of always the tail; auto-stick to the tail while streaming, release to manual once the user scrolls.

(The guideline's "Page flipping" pattern with pre-pagination at ~400–500 char boundaries applies — though here `textContainerUpgrade` can stay flicker-free since the layout doesn't change.)

## 🟢 U7 — `BODY_MAX = 400` likely overestimates the body (`views.ts`)

**Severity:** Low — tuning.

The guideline says **~400–500 chars fill the *full* screen** (576×288). The body is only `200/288 ≈ 70%` of the height, so realistic capacity is closer to **~280–340 chars**, not 400. At 400 the tail may clip at the bottom edge. Also `.slice(-BODY_MAX)` cuts mid-word and ignores `\n`. Tune `BODY_MAX` against the device and consider a word-/newline-aware slice. The comment already flags "tuned later on device" — keep that promise.

## 🟢 U8 — Header is static (`render.ts`)

**Severity:** Low.

The header container is created with `content: "Hermes"` and never updated. It's a natural place to surface the **active session title** (or a connection indicator) once M3 lands. Consider rendering `state.sessions.active` title into the header in `renderChat`.

## 🟢 U9 — Leftover Vite demo files (repo)

**Severity:** Low — cleanliness.

`src/counter.ts`, `src/style.css`, and `src/assets/` (hero.png, vite.svg, typescript.svg) are scaffolding from the Vite starter and are no longer imported by `main.ts`. They don't affect the glasses (which render containers, not HTML/CSS), but they add noise and may confuse `pack`. Remove once confirmed unreferenced by `index.html`.

## 🟢 U10 — Sessions view must rebuild, not upgrade (M3, future)

**Severity:** Low now (not built) — but a guaranteed bug if implemented naively.

Two guideline rules collide with the obvious implementation:
1. *"Lists cannot be updated in-place — the entire page must be rebuilt to change list content."*
2. `createStartUpPageContainer` is **one-shot** (already spent by `buildChatPage`).

So the M3 sessions view must be drawn with **`rebuildPageContainer`**, and any change to the session list (new session, reorder) requires another `rebuildPageContainer` — not a `textContainerUpgrade`/list-mutation. The sessions list container also needs its own `isEventCapture: 1` while shown. This duplicates `PLAN-REVIEW` #2 and is restated here so the UI implementer sees it in context.

## Recommended order of fixes

1. **U2** (silent-fail upgrade params) — one-line-ish fix, unblocks reliable rendering on device.
2. **U1** (render queue + coalesce) — required before any real streaming test; pairs with U4.
3. **U4** (BLE timeout) — small, prevents wedges; fold into the same render-pipeline change.
4. **U5** (cleanup) — add now so M4's mic work inherits it.
5. **U3 / U6** — verify glyphs; add scrollback.
6. **U7–U10** — tuning, polish, and the M3 reminder.
